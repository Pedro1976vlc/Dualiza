7.2.- Objetos Kubernetes.

Los objetos de Kubernetes son entidades persistentes dentro del sistema de Kubernetes. Kubernetes utiliza estas entidades para representar el estado de tu clúster. Indican las aplicaciones, los recursos disponibles para estas y las políticas de comportamiento como las políticas de reinicio o tolerancia a fallo. Cada objeto posee dos campos:

  • spec: Obligatorio, indica el estado deseado del objeto.
  • status: Estado actual del objeto.

Kubernetes intenta conseguir crear y mantener todos los objetos indicados en la especificación, en caso de no poder cumplir, tomará medidas para que se cumpla.

Para la descripción de un objeto se puede usar el comando kubectl pasándole un fichero en formato YAML, realizando las llamadas oportunas a la API. 

Los campos obligatorios del fichero son:

  • apiVersion. Qué versión de la API de Kubernetes estás usando para crear este objeto. 
  • kind. Tipo de objeto, puede ser un de tipo pod, cronjob ,demonio, despliegue, servicios, volúmenes o réplica entre otros.
  • metadata. Datos para caracterizar al objeto e identificarlo.
  • spec. Depende de cada objeto definido en kind. En la siguiente URL se pueden consultar los atributos de cada uno de ellos: https://kubernetes.io/docs/reference/generated/kubernetes-api/v1.21/#pod-v1-core.

A continuación, se muestra un fichero YAML de ejemplo en el que se aprecian  los campos requeridos y la spec del objeto Deployment de Kubernetes:

apiVersion: apps/v1 # Usa apps/v1beta2 para versiones anteriores a 1.9.0
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  selector:
    matchLabels:
      app: nginx
  replicas: 2 # indica al controlador que ejecute 2 pods
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.7.9
        ports:
        - containerPort: 80

Vídeo explicativo donde se detallan cuales son características de los principales Objetos de Kubernetes.

Los objetos básicos de Kubernetes son:

  • Pods: Tal y como se ha definido anteriormente un grupo de contenedores con una especificación y unos recursos.
  • Services: Permite exponer una aplicación. Se indica el o los pods a los que hace referencia y la forma de acceder a los mismos. Un ejemplo de configuración extraído de la documentación oficial. https://kubernetes.io/docs/concepts/services-networking/service/#defining-a-service.
apiVersion: v1
kind: Service
metadata:
&nbsp; name: my-service
spec:
&nbsp; selector:
&nbsp; &nbsp; app: MyApp
&nbsp; ports:
&nbsp; &nbsp; - protocol: TCP
&nbsp; &nbsp; &nbsp; port: 80
&nbsp; &nbsp; &nbsp; targetPort: 9376

El atributo spec.selector.app tiene el valor MyApp, refiriendose al Pod que hace visible.

  • Volume: Permite el almacenamiento persistente independiente de los Pods, posee una gran variedad de tipos como awsElasticBlockStore, azureDisc, Cephfs o OpenStack.
  • Namespaces: Cluster virtualizados, pensado para entornos con muchos usuarios y muchos proyectos, son una forma de asignar los recursos del clúster entre varios usuarios.

Para saber más

Para la gestión de objetos K8s se utiliza el comando kubectl (Kubernetes CLI), ya sea usando parámetros o a través de ficheros YAML. Este libro (https://kubectl.docs.kubernetes.io/) se centra en el uso de Kubectl para administrar aplicaciones de forma declarativa en Kubernetes, su lectura es muy recomendada para tener una visión amplia del comando kubectl y sus distintas posibilidades y funciones.
