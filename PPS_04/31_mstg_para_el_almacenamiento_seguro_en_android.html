3.1.- MSTG para el almacenamiento seguro en Android.

La guía de pruebas de seguridad móvil expone los fundamentos teóricos, técnicas y características de almacenamiento en Android.

Establece 11 test de pruebas de almacenamiento  y 1 de plataforma.

MSTG-STORAGE-1:

 Los sistemas de almacenamiento de credenciales que proporciona el sistema se han de utilizar para almacenar datos confidenciales.

MSTG-STORAGE-2:

No se deben almacenar datos confidenciales fuera del contenedor de la aplicación o de las instalaciones de almacenamiento de credenciales del sistema.

MSTG-STORAGE-3:

No se escriben datos confidenciales en los registros de la aplicación.

MSTG-STORAGE-4: 

No se comparte ningún dato sensible con terceros a menos que sea una parte necesaria de la arquitectura.

MSTG-STORAGE-5:

La caché del teclado está deshabilitada en las entradas de texto que procesan datos confidenciales.

MSTG-STORAGE-6:

No se exponen datos sensibles a través de mecanismos de IPC.

MSTG-STORAGE-7:

 No se exponen datos confidenciales, como contraseñas o pines, a través de la interfaz de usuario.

MSTG-STORAGE-8:

 No se incluyen datos confidenciales en las copias de seguridad generadas por el sistema operativo móvil.

MSTG-STORAGE-9:

 La aplicación elimina los datos confidenciales de las vistas cuando se mueven al fondo.

MSTG-STORAGE-10: 

La aplicación no retiene datos confidenciales en la memoria más de lo necesario y la memoria se borra explícitamente después de su uso.

MSTG-STORAGE-11:

La aplicación aplica una política mínima de seguridad de acceso al dispositivo, como exigir al usuario que establezca un código de acceso para el dispositivo.

MSTG-PLATFORM-2:

Todas las entradas de fuentes externas y del usuario se validan y, si es necesario, se desinfectan. Esto incluye datos recibidos a través de la interfaz de usuario, mecanismos de IPC como intenciones, URL personalizadas y fuentes de red.

Estas MSTG se asocian a los diferentes requerimientos MASVS para la certificación de las aplicaciones en los niveles 1 y 2, en el caso de almacenamiento (capítulo 2) existe una correspondencia directa entre el requerimiento y el caso de prueba.

Guía OWASP de pruebas de seguridad para móviles. Se trata de un documento que aporta información específica y técnica sobre los controles de seguridad que se deben contemplar a la hora de realizar tareas de pentesting en aplicaciones móviles.

El Estándar OWASP de Verificación de Seguridad de Aplicaciones Móviles, es un esfuerzo comunitario para establecer un marco de requisitos de seguridad necesarios para diseñar, desarrollar y probar aplicaciones móviles seguras en iOS y Android.

Shared Preferences:

En común con muchas aplicaciones, no solo para dispositivos móviles. Se tiene la necesidad de almacenar información en formato clave-valor, es decir se asocia un nombre denominado clave a un valor que puede ser una cadena de texto o un valor numérico. Dada su utilidad muchos sistemas, lenguajes y librerías ofrecen interfaces que facilitan el almacenamiento, por ejemplo los ficheros .properties de Java en el que se almacenan pares clave-valor. En el caso de Android se almacena en XML y establece algunas medidas de seguridad como que solo la aplicación propietaria maneje el fichero.

Ejercicio Resuelto

Un ejemplo de uso para Kotlin, en el caso de preferencias compartidas:

var sharedPref = getSharedPreferences("key", Context.MODE_WORLD_READABLE)
var editor = sharedPref.edit()
editor.putString("username", "administrator")
editor.putString("password", "supersecret")
editor.commit()

Se caracteriza por una perfecta combinación de características claramente orientadas a la funcionalidad durante la programación, centrándose en la seguridad, la claridad y la interoperabilidad. Está diseñado para ser un lenguaje de programación orientado a objetos de calidad industrial, y para ser un lenguaje mejor que Java pero todavía ser plenamente interoperable con código Java, permitiendo a las compañías hacer una migración gradual de Java a Kotlin.

Bases de datos SQLite:

Este es un sistema gestor de base de datos sencillo y mínimo, que incluye la funcionalidad de la SGBD en la librería o paquete de la aplicación, almacenando la estructura y los datos en un simple fichero de texto. En contraprestación a los pocos recursos que necesita no ofrece toda la potencia de una SGBD completa como MySql o Postgre, limitando por ejemplo los tipos de datos de los campos o las posibilidades de gestión de usuarios. 

Las recomendaciones para usuarios y contraseñas son las mismas que en el caso de las aplicaciones web. codificación usando los diferentes métodos tratados en el tema anterior, pero el acceso al resto de los datos es tan sencillo como acceder al fichero que los contiene, normalmente se encuentra en al ruta:

/data/data/<package-name>/databases/

Una aplicación maliciosa o el acceso físico al dispositivo compromete la seguridad de toda la información contenida.

La solución que propone OWASP es la de codificar el fichero de forma que aunque se tenga acceso al mismo no sea posible la obtención de los datos, recomendado el uso de la librería SQLCipher.

Esta librería extiende a SQLite añadiendo protección a los ficheros de base de datos de forma automática incluyendo un usuario y contraseña para ello, estas credenciales se pueden:

  • Solicitar cada vez que el usuario accede a la aplicación.
  • Almacenarlas en un servidor para que la aplicación las solicite al iniciarse.

Bases de datos de Firebase y Realm Database:

Dos alternativas para almacenamiento de datos en en Android. La primera forma parte de una plataforma  de desarrollo de aplicaciones web y móviles de Google siendo Realtime database uno de los servicios que ofrece es almacenamiento en tiempo real en formato JSON en la nube a través de una API Rest, además de implementar el protocolo SSE para recibir eventos del servidor. La segunda, Realm Database es orientada a objetos y diseñada específicamente para sistemas operativos móviles.

Los ficheros se encuentran codificados, y es posible almacenar la clave tanto en un fichero de configuración en un servidor remoto. El uso es transparente para el desarrollador.

Protocolo que describe los mecanismos necesarios para enviar información asíncrona desde el servidor a los clientes. Formando parte del estándar HTML5 del W3C.

Almacenamiento interno:

En este caso el sistema de almacenamiento es el clásico para un sistema de ficheros, en teoría sería posible que cualquier aplicación pudiera acceder a los ficheros, pero Android establece un parámetro en el constructor de las clases que manejan ficheros en el que se indica que en acceso es privado con la constante Context.MODE_PRIVATE.

Almacenamiento externo:

El sistema no provee de mecanismo de protección ante datos almacenados en dispositivos externos, y cualquier aplicación o en caso de ser extraíble, otro sistema externo puede acceder a los datos.

KeyStore:

La mayor parte de los sistemas operativos actuales poseen almacenes protegidos con credenciales en los que se almacenan contraseñas y claves criptográficas y certificados. Android no es una excepción y define KeyStore con este fin. Para el acceso a este almacén se ha de facilitar unas credenciales por parte de usuario, como es la huella, el patrón o el PIN de desbloqueo del dispositivo.

Las aplicaciones definen cómo se accede a las claves que estas generan o importan, el acceso estas se otorga por:

  • Criptografía: Conjunto de algoritmos criptográficos en los que se puede usar la clave:
  • Validez temporal: Se define el tiempo en que el usuario puede utilizar las claves.
  • Autenticación del usuario: Solo se puede usar si el usuario se autenticó hace poco tiempo.
