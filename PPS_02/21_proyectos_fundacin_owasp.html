2.1.- Proyectos fundación OWASP.

La fundación posee decenas de proyectos para el desarrollo de aplicaciones seguras, entre todos ellos destacan:

Top Ten Web Application Security Risk.

Listado de los 10 riesgos más importantes y comunes en el desarrollo web. Este listado se actualiza cada pocos años, se añaden nuevos riesgos, desaparecen otros, o cambia su relevancia o posición en el listado. La versión actual es del 2020. Se puede consultar en:

https://owasp.org/www-project-top-ten/

Top Ten Mobile Security Risk.

Equivalente a la anterior, pero para aplicaciones móviles. La última versión es la del 2016.

Dependency-Trak

Plataforma para el análisis inteligente de componentes, que permite minimizar el riesgo. Las aplicaciones actuales se construyen basándose en otros componentes y librerías sobre las que no se tiene control. Con esta herramientas es posible realizar una monitorización de los diferentes componentes. Utiliza los repositorios de código y gestores de paquetes utilizados en los lenguajes más usados, como: npm, maven o composer. La página web de proyecto es:

https://docs.dependencytrack.org/

Juice Shop.

Aplicación plagada de errores de seguridad útil para el aprendizaje y entrenamiento en uso de herramientas de seguridad. Desarrollada con Node, Servicios REST con Express y Angular. Se puede descargar el código o utilizar una imagen de Docker-OWASP. Existe un libro oficial sobre la aplicación.

Mobile Security Testing Guide.

Guía de desarrollo y pruebas de seguridad en aplicaciones móviles. Es uno de los proyectos de mayor relevancia de la fundación, se trata en mayor profundidad en posteriores puntos de la unidad.

Web Security Testing Guide.

Guía completa para probar la seguridad de las aplicaciones web y los servicios web, proporcionando un marco con las mejores prácticas en las aplicaciones web basada en escenarios. En la primera parte de la guía se explica la importancia de las pruebas, los tipos, desarrollo del plan de pruebas y proceso de pruebas Además, establece un marco de trabajo para especificar, diseñar y realizar las pruebas. También especifica un gran conjunto de pruebas a llevar a cabo para medir la seguridad de una aplicación, agrupadas en tipos como fuga de información (relacionado con las fuentes abiertas), manejo de sesiones o validación de entradas.

En cada caso de pruebas se  indica el problema de seguridad de forma técnica, y la forma de realizar las pruebas:

  • Comandos como curl.
  • Pruebas de caja blanca.
  • Pruebas de caja negra.
  • Pruebas de Integración
  • Pruebas Fuzzy.
  • Pruebas E2E.

Zed Attack Proxy (ZAP)

Herramienta para el testeo de aplicaciones web, permite manipular el tráfico que pasa a través del mismo. Algunas de las funcionalidades que ofrece son:    

  • Proxy de interceptación: Permite ver todo el tráfico entre el navegador y el servidor web. Puede modificar el tráfico HTTP en ambas direcciones de la comunicación (entre el servidor web y el navegador).
  • Spider: Ayuda a descubrir nuevas URL’s en el sitio auditado. Una de las maneras que realiza esto es analizando el código HTML de la página para descubrir etiquetas <a> y seguir sus atributos href.
  • Forced Browsing: Intenta descubrir directorios y archivos no indexados en el sitio como pueden ser páginas de inicio de sesión.
  • Active Scan: Genera diferentes ataques web, como CSRF, XSS, Inyección SQL.
  • Pruebas Fuzzer: Proporcionar datos inválidos, inesperados o aleatorios con el objetivo de supervisar y vigilar el comportamiento del sistema.

OWASP ModSecurity Core Rule Set.

Sistema de detección de ataques basado en cortafuegos para aplicaciones web. En el proyecto se incluyen diferentes tutoriales: instalación, configuración de Apache, IIS y Mginx, definición de reglas y disparadores entre otros. El proyecto ha crecido y posee su propia página web fuera de la fundación.

https://coreruleset.org/

Se basa en la definición de reglas, por ejemplo la siguiente bloquea la IP de un spammer

SecRule TX:block_spammer_ip "@eq 1" \
   "msg:'HTTP Blacklist match for spammer IP',\
   severity:'CRITICAL',\
   id:910160,\
   phase:request,\
   block,\
   t:none,\
   tag:'application-multi',\
   tag:'language-multi',\
   tag:'platform-multi',\
   tag:'attack-reputation-ip',\
   chain,\
   skipAfter:END_RBL_CHECK"
        SecRule TX:httpbl_msg "(?i)^.*? spammer .*?$" \
           "setvar:'tx.msg=%{rule.msg}',\
           setvar:tx.anomaly_score=+%{tx.critical_anomaly_score},\
           setvar:tx.%{rule.id}-AUTOMATION/MALICIOUS-%{matched_var_name}=%{matched_var},\
           setvar:ip.reput_block_flag=1,\
           expirevar:ip.reput_block_flag=%{tx.reput_block_duration},\
           setvar:'ip.reput_block_reason=%{rule.msg}',\
           setvar:ip.previous_rbl_check=1,\
           expirevar:ip.previous_rbl_check=86400"

Además, es muy importante destacar que existen grupos locales, repartidos por todo el mundo, especializados en ofrecer cursos y conferencias. En España existen dos, uno en Bilbao y otro en Sevilla, siendo este último el que más actividad reciente ha tenido; y en el que se puede encontrar diferentes presentaciones y documentos. La página oficial del grupo de Sevilla es:

https://owasp.org/www-chapter-sevilla/

Introducción a la fundación OWASP y los proyectos más destacados.

El fuzzing o fuzz testing consiste en pruebas semiautomáticas o automáticas donde se proveen datos al azar, inválidos y no esperados en la sección de entrada de datos del software.

Probar flujos de ejecución en una aplicación de comienzo a fin. se prueban escenarios que combinan diferentes casos de uso/funcionalidades tal cual como lo haría un usuario.

Comando disponible en la mayoría de los sistemas basados ​​en UNIX. Es una abreviatura de “Client URL”. Los comandos de Curl están diseñados para funcionar como una forma de verificar la conectividad a las URL. Son una gran herramienta para transferir datos.
