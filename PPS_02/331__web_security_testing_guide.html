3.3.1. - Web Security Testing Guide.

OWAPS establece a partir del estándar de verificación de aplicaciones una guía específica para aplicaciones y servicios  web, la dirección del proyecto es:

https://owasp.org/www-project-web-security-testing-guide/

Actualmente la se encuentra en la versión 4.2, disponible online en https://owasp.org/www-project-web-security-testing-guide/v42/ y en formato pdf. Se estructura en escenarios agrupados en categorías, en concreto 12:

  • Information Gathering.
  • Configuration and Deployment Management Testing.
  • Identity Management Testing.
  • Authentication Testing.
  • Authorization Testing.
  • Session Management Testing.
  • Input Validation Testing.
  • Error Handling.
  • Cryptography.
  • Business Logic Testing.
  • Client Side Testing.
  • API Testing.

Estos escenarios exponen los fundamentos técnicos a evaluar en las pruebas de penetración, las cuales se han de reproducir para comprobar la seguridad del sistema. Cada escenario  se identifica con las letras WSTG-<categoria>--<numero>, la categoría se nombra con 4 caracteres que indican el tipo de prueba o debilidad, algunos ejemplos de categorías:

  • INFO: Recopilación de información.
  • CONF: Comprobación de configuraciones de elementos.
  • APIT: Evaluación de API.
  • ATHN: Autenticación.

Descripción textual alternativa

Introducción al proyecto WSTG describiendo el documento de referencia sobre las pruebas.

Se dispone también de una libro Excel que permita realizar la verificación del resultado de la evaluación de cada escenario en un proyecto, esta hoja se encuentra en:

https://github.com/OWASP/wstg/tree/master/checklist

Incluye cuatro hojas de cálculo:

  • Testing Checklist: la primera con las categorías, en cada fila se tiene el escenario, en nombre de la prueba, los objetivos a conseguir, el estado y una celda para notas.
  • Summary Findings: Fallos de seguridad encontrados, en el que se indica el nombre, el escenario, el elemento afectado (url..), la probabilidad de ataque, el nivel de riesgo, observaciones, las recomendaciones y el test en que se detecto.
  • Calculadora de evaluación de riesgos: A partir del tipo de aplicación y sus características se establece el nivel de riesgo de la aplicación se obtiene el nivel de riesgo de sufrir un ataque de la aplicación. Tiene en cuenta 2 factores:
    • Probabilidad de ataque: Tiene en cuenta los posibles elementos vulnerables como la dificultad de detección de intrusión  y los factores que determinan las posibles amenazas como el nivel de habilidades requerido para romper el sistema.
    • Factores de impacto: Se evalúa los factores técnicos como la pérdida de cofidencialidad y los factores de negocio como la pérdida de reputación ante por ejemplo fuga de información.
  • Referencias: Equivalencia entre los cuadro desplegables de la calculadora y su valor numérico, solo útil para cálculos internos.

Ataque a un sistema que pretende de encontrar las debilidades de seguridad en especial sus datos.

Debes conocer

Test de penetración de Incibe.

Descripción textual alternativa.

Concepto de test de penetración usado por OWASP en la verificación del nivel de aplicaciones web.

Cada uno de estos escenarios responde a un tipo de amenaza o vulnerabilidad, estas pueden ser muy diferentes entre ellas, por ejemplo los fundamentos tecnológicos afectados (lado del cliente, lado del servidor, lenguaje de programación...), las causas que originan la amenaza determinando las herramientas a utilizar en los test de penetración. Por ejemplo el escenario WSTG-CONF-06 Test HTTP Methods.

Analiza los diferentes métodos del protocolo HTTP haciendo referencia al RFC en el que se definen, indicado además que la mayor parte de las aplicaciones web solo necesitan los metodos GET y POST, aunque en servicios REST también se utilizan PUT y DELETE.

En el siguiente punto indica los objetivos que persigue el test de penetración:

  • Indicar los métodos HTTP que implementa la aplicación.
  • Evaluar el cumplimento del control de acceso.
  • Comprobar la existencia de vulnerabilidades de tipo XST.
  • Probar la anulación de los métodos no necesarios.

El punto siguiente en todos los escenarios indica como realizar las pruebas, este dependerá de cada escenario concreto e indicara las técnicas y herramientas necesarias para su realización, en este caso nmap, ncat o curl.
