2.2.- OWASP Top Ten Web.

El software lleva muchas décadas utilizándose instalado en estaciones de trabajo, en especial para un uso doméstico y en pequeñas y medianas empresas, por ejemplo en programas ofimáticos. En la última década se ha producido un movimiento desde las aplicaciones nativas, destacando las del sistema operativo Windows hacia versiones que utilizan el navegador y la web, como Office 365 o Autocad, alguno de ellos con prácticamente la misma funcionalidad que sus versiones nativas. También los organismos oficiales y bancos han creado aplicaciones web para interactuar con sus clientes y ciudadanos. Junto con el aumento de uso de las aplicaciones web el interés por los hackers en este tipo de software ha crecido al mismo o mayor ritmo.

La fundación OWASP recolecta los 10 mayores vulnerabilidades y riesgos a partir de las contribuciones de empresas relacionadas con la seguridad de aplicaciones proporcionando técnicas básicas para protegerse. Cada cierto tiempo la lista se actualiza con nuevos riesgos, la última versión es la del 2017 que ha tenido en cuenta los nuevos lenguajes de servidor, los microservicios, los servicios REST, las aplicaciones de una sola página (SPA) o los frameworks de aplicaciones web del lado del cliente como Angular, Vue o en CSS Boostrap (aunque utiliza Javascript y jquery entre otras).
También incluye una serie de sugerencias para perfiles concretos del desarrollo web como desarrolladores, testers o administradores de aplicaciones.

OWASP Top Ten Web es un ranking de los diez riesgos y vulnerabilidades de seguridad de la información más peligrosos para aplicaciones web:

1. Inyección (Injection).

Las fallas de inyección, como la inyección de SQL, NoSQL, OS y LDAP, ocurren cuando se envían datos que no son de confianza a un intérprete como parte de un comando o consulta. Los datos hostiles del atacante pueden engañar al intérprete para que ejecute comandos no deseados o acceda a datos sin la debida autorización.

2. Autenticación rota (Broken Authentication).

Las funciones de la aplicación relacionadas con la autenticación y la administración de sesiones a menudo se implementan de manera incorrecta, lo que permite a los atacantes comprometer contraseñas, claves o tokens de sesión, o aprovechar otras fallas de implementación para asumir las identidades de otros usuarios de forma temporal o permanente.

3. Exposición de datos sensibles (Sensitive Data Exposure).

Muchas aplicaciones web y API no protegen adecuadamente los datos confidenciales, como los financieros o la información médica. Los atacantes pueden robar o modificar esos datos débilmente protegidos para cometer fraude con tarjetas de crédito, robo de identidad u otros delitos. Los datos confidenciales pueden verse comprometidos sin protección adicional, como el cifrado en reposo o en tránsito, y requieren precauciones especiales cuando se intercambian con el navegador.

4. Entidades externas XML (XML External Entities, XXE).

 Muchos procesadores XML más antiguos o mal configurados evalúan las referencias de entidades externas dentro de los documentos XML. Se pueden utilizar entidades externas para divulgar archivos internos, escaneo de puertos internos, ejecución remota de código y ataques de denegación de servicio.

5. Control de acceso roto (Broken Access Control).

Las restricciones sobre lo que pueden hacer los usuarios autenticados a menudo no se aplican correctamente. Los atacantes pueden aprovechar estos defectos para acceder a funciones y / o datos no autorizados, como acceder a las cuentas de otros usuarios, ver archivos confidenciales, modificar los datos de otros usuarios, cambiar los derechos de acceso, etc.

6. Configuración incorrecta de seguridad (Security Misconfiguration).

La mala configuración de seguridad es el problema más común. Esto suele ser el resultado de configuraciones predeterminadas inseguras, configuraciones incompletas o ad hoc, almacenamiento en la nube abierta, encabezados HTTP mal configurados y mensajes de error detallados que contienen información confidencial. No solo todos los sistemas operativos, marcos, bibliotecas y aplicaciones deben estar configurados de manera segura, sino que también deben ser parcheados / actualizados de manera oportuna.

7. Secuencias de comandos entre sitios (Cross-Site Scripting, XSS).

Los defectos de XSS ocurren cuando una aplicación incluye datos que no son de confianza en una nueva página web sin la validación o el escape adecuados, o actualiza una página web existente con datos proporcionados por el usuario mediante una API de navegador que puede crear HTML o JavaScript. XSS permite a los atacantes ejecutar scripts en el navegador de la víctima que pueden secuestrar sesiones de usuario, desfigurar sitios web o redirigir al usuario a sitios maliciosos.

8. Deserialización insegura (Insecure Deserialization).

La deserialización insegura a menudo conduce a la ejecución remota de código. Incluso si las fallas de deserialización no dan como resultado la ejecución remota de código, se pueden usar para realizar ataques, como por ejemplo: ataques de reproducción, ataques de inyección y ataques de escalada de privilegios.

9. Uso de componentes con vulnerabilidades conocidas (Using Components with Known Vulnerabilities).

Los componentes, como bibliotecas, marcos y otros módulos de software, se ejecutan con los mismos privilegios que la aplicación. Si se explota un componente vulnerable, dicho ataque puede facilitar la pérdida de datos o la toma de control del servidor. Las aplicaciones y API que utilizan componentes con vulnerabilidades conocidas pueden socavar las defensas de las aplicaciones y permitir varios ataques e impactos.

10. Registro y monitoreo insuficientes (Insufficient Logging & Monitoring).

El registro y la supervisión insuficientes, junto con una gestión ineficaz de las respuestas ante incidentes, permite a los atacantes atacar aún más los sistemas, mantener la persistencia, introducir cambios en los sistemas y manipular, extraer o destruir datos. La mayoría de los estudios demuestran que el tiempo para detectar una infracción es superior a 200 días, generalmente detectados por agentes externos en lugar de procesos internos o monitoreo.

Deficiencias de seguridad de un sistema informático que pueden permitir el acceso a usuarios no autorizados y la realización de acciones maliciosas intencionadas.

NoSQL (Not Only SQL). Sistema de gestión de base de datos que difieren del modelo de bases de datos relacionales. Este tipo de almacenamiento de datos relaja las restricciones inherentes al propio modelo relacional. Este tipo de bases de datos no suelen proveer garantías ACID, pero resultan mucho más eficientes y baratas para manipular datos sin tener necesariamente que adaptarlos a una estructura rígida. Son ideales para almacenar datos de manera masiva, lo que responde muy bien a la enorme riqueza de datos que genera la Web, y analizarlos sin seguir necesariamente unos estándares que no necesariamente se adaptan a ellos.

LDAP (Lightweight Directory Access Protocol) Protocolo a nivel de aplicación que permite el acceso a un servicio de directorio ordenado y distribuido para buscar diversa información en un entorno de red.
