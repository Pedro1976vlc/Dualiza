2.3. -OWASP Top Ten Móvil.

Si bien las aplicaciones web son las que están experimentando una mayor proliferación en los últimos tiempos, las aplicaciones nativas para dispositivos móviles también han crecido exponencialmente; siendo los teléfonos móviles los dispositivos electrónicos más utilizados hoy en día, por delante de los clásicos equipos PC de sobremesa y ordenadores portátiles; y por tanto, se han convertido en el principal objetivo de los ciberdelincuentes.



De igual forma que las aplicaciones web se ha definido un listado con las 10 categorías de riesgos más peligrosos en el desarrollo de aplicaciones móviles, el listado actual es la versión del 2016, la página en la que se puede encontrar es:

https://owasp.org/www-project-mobile-top-10

Reflexiona

En caso de ser un hacker que persigue un objetivo económico ¿hacia qué tipo de aplicaciones te dirigirías web o nativas para móviles? ¿Y si se desea robar datos personales y escuchar las conversaciones?

Para cada categoría se indica la situación de riesgo o el vector de ataque, el grado de explotabilidad, la prevalencia, la detectabilidad y el impacto que puede tener. Se explica también los elementos implicados y posibles soluciones para evitar esa categoría de riesgos.

1. Uso impropio de la plataforma.

M1: Improper Platform Usage.

Hace referencia al uso indebido de una función de la plataforma o la falta de uso de los controles de seguridad de la plataforma. Para aprovechar esta vulnerabilidad, la organización debe exponer un servicio web o una llamada a la API que consume la aplicación móvil. El servicio expuesto o la llamada a la API se implementa utilizando técnicas de codificación inseguras que producen una vulnerabilidad en el servidor. A través de la interfaz móvil, un atacante puede enviar entradas maliciosas o secuencias inesperadas de eventos al punto final vulnerable.

Hay varias formas en que las aplicaciones móviles pueden experimentar este riesgo:

  1. Violación de las pautas publicadas. Todas las plataformas tienen pautas de desarrollo para la seguridad (p.e, Android, iOS, Windows Phone. Si una aplicación contradice las recomendaciones del fabricante, estará expuesta a este riesgo. 
  2. Violación de convención o práctica común. No todas las mejores prácticas están codificadas en la guía del fabricante. En algunos casos, existen mejores prácticas de facto que son comunes en las aplicaciones móviles.
  3. Mal uso involuntario. Algunas aplicaciones tienen la intención de hacer lo correcto, pero en realidad se equivocan en una parte de la implementación. Esto podría ser un error simple, como configurar el indicador incorrecto en una llamada a la API, o podría ser un malentendido sobre cómo funcionan las protecciones. Las fallas en los modelos de permisos de la plataforma se incluyen en esta categoría. 

2. Almacenamiento de datos inseguros.

M2: Insecure Data Storage.

Un atacante que robe o adquiera físicamente un dispositivo móvil, puede conectarlo a una computadora  y mediante las herramientas software adecuadas, que en muchos casos están  disponible gratuitamente; puede todos los directorios de aplicaciones de terceros que contienen información de identificación personal (PII) almacenada u otros activos de información confidencial. 

3. Comunicaciones Inseguras.

M3: Insecure Communication.

Cuando se diseña una aplicación móvil, los datos se intercambian comúnmente entre un  cliente y un servidor. Cuando la solución transmite sus datos, debe atravesar la red del operador del dispositivo móvil e Internet. Los atacantes pueden aprovechar las vulnerabilidades para interceptar datos confidenciales mientras viajan.  Las aplicaciones móviles con frecuencia no protegen el tráfico de la red. Pueden usar SSL / TLS durante los procesos de  autenticación, pero no en otros lugares. Esta incoherencia conlleva el riesgo de exponer los datos y los identificadores de sesión a la interceptación. 

4. Autenticación Insegura.

M4: Insecure Authentication.

Los agentes que aprovechan las vulnerabilidades de autenticación normalmente lo hacen empleando herramientas disponibles o personalizadas que permiten automatizar los ataques. Hay muchas formas diferentes en que una aplicación móvil puede sufrir una autenticación insegura: 

  1. Si una aplicación móvil almacena contraseñas o secretos compartidos localmente en el dispositivo, lo más probable es que sufra una autenticación insegura;
  2. Si la aplicación móvil usa una política de contraseña débil para simplificar el ingreso de una contraseña, sufre de autenticación insegura; o
  3. Si la aplicación móvil utiliza una función como TouchID, sufre de autenticación insegura.

Lo deseable es que la información de autenticación que se guarda localmente en el móvil, se almacena encriptada empleando claves de cifrado fuertes.

5. Criptografía insuficiente.

M5: Insufficient Cryptography. 

Aprovechando esta debilidad, un adversario puede devolver con los datos cifrados o confidenciales a su forma original. Los principales fallos que pueden provocar esta situación son:

  1. Incluir las claves en el mismo directorio legible por el atacante que el contenido cifrado;
  2. Poner las claves a disposición del atacante;
  3. Las claves se pueden interceptar mediante ataques binarios. 
  4. No hay manera más fácil de manejar mal el cifrado, móvil o de otro tipo, que crear y usar sus propios algoritmos o protocolos de cifrado.

Para manejar datos confidenciales, las principales recomendaciones son:

  1. Siempre que sea posible, se debe evitar el almacenamiento de datos confidenciales en  el dispositivo móvil siempre que sea posible.
  2. Es muy recomendable utilizar algoritmos criptográficos  de última generación que hayan sido aceptados como sólidos por la comunidad de seguridad. 
  3. Seguir las pautas marcadas por el NIST (National Institute of Standards and Technology) sobre algoritmos recomendados

6. Autenticación insegura.

M6: Insecure Authorization.

Los adversarios pueden realizar ataques binarios contra la aplicación móvil e intentar ejecutar una funcionalidad privilegiada que solo debería ser ejecutable con un usuario de mayor privilegio mientras la aplicación móvil está en modo 'fuera de línea'.

7. Calidad de código deficiente.

M7: Poor Code Quality.

Los problemas de calidad del código son bastante frecuentes en la mayoría de los códigos móviles. Por lo general, es difícil detectar este tipo de problemas mediante la revisión manual del código. En su lugar, los atacantes utilizan herramientas de terceros que realizan análisis estático. Este tipo de herramientas normalmente identificarán pérdidas de memoria, desbordamientos de búfer y otras malas prácticas de programación. Los piratas informáticos con conocimientos y experiencia de muy bajo nivel pueden explotar eficazmente este tipo de problemas. El objetivo principal típico es ejecutar código externo dentro del espacio de direcciones del código móvil.

8. Manipulación de código. 

M8: Code Tampering.

Un atacante aprovechará la modificación del código a través de formas maliciosas de las aplicaciones alojadas en tiendas de aplicaciones de terceros. El atacante también puede engañar al usuario para que instale la aplicación mediante ataques de phising.  Las formas modificadas de aplicaciones son sorprendentemente más comunes de lo que se cree. Existe toda una industria de seguridad construida alrededor de la detección y eliminación de versiones no autorizadas de aplicaciones móviles dentro de las tiendas de aplicaciones. 

9. Ingeniería inversa.

M9: Reverse Engineering.

Un atacante descarga la aplicación objetivo de una tienda de aplicaciones y la analiza dentro de su propio entorno local utilizando un conjunto de herramientas diferentes (como por ejemplo: IDA Pro, Hopper, otool, strings, Ghidra o BinaryNinja). Generalmente, todo el código móvil es susceptible de ingeniería inversa. Para evitar una ingeniería inversa eficaz, debe utilizar una herramienta de ofuscación. Existen muchos ofuscadores gratuitos y comerciales en el mercado. 

10. Funcionalidades Extrañas. 

M10: Extraneous Functionality

Por lo general, un atacante busca comprender la funcionalidad extraña dentro de una aplicación móvil para descubrir una funcionalidad oculta en los sistemas backend. A menudo, los desarrolladores incluyen funciones ocultas de puerta trasera u otros controles de seguridad de desarrollo internos que no están destinados a ser lanzados a un entorno de producción.
