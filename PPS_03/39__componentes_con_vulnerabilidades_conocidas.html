3.9. - Componentes con vulnerabilidades conocidas.

Las aplicaciones y API que utilizan componentes con vulnerabilidades conocidas pueden debilitar las defensas de las aplicaciones y permitir diversos ataques. Si se explota un componente vulnerable, el ataque puede provocar una pérdida de datos o facilitar la toma de control del servidor. 

Algunas ejemplificaciones de situaciones que ilustran el uso de vulnerabilidades conocidas son:

  • El software es vulnerable, no posee soporte o se encuentra desactualizado. Esto incluye el sistema operativo, servidor web o de aplicaciones, DBMS, APIs y todos los componentes, ambientes de ejecución y bibliotecas.
  • No se analizan los componentes periódicamente ni se realiza seguimiento de los boletines de seguridad de los componentes utilizados.
  • No se parchea o actualiza la plataforma subyacente, frameworks y dependencias, con un enfoque basado en riesgos. Esto sucede comúnmente en ambientes en los cuales la aplicación de parches se realiza de forma mensual o trimestral bajo control de cambios, lo que deja a la organización expuesta innecesariamente durante varios días o meses de exposición a vulnerabilidades ya solucionadas.
  • No se realiza adecuadamente la configuración correcta de seguridad de los componentes.

Sistema gestor de base de datos (SGBD o DBMS), es un conjunto de programas que permiten el almacenamiento, modificación y extracción de la información en una base de datos.

Cada organización debe asegurar la existencia de un plan para monitorear, evaluar y aplicar actualizaciones o cambios de configuraciones durante el ciclo de vida de las aplicaciones. Algunas de la principales acciones que se pueden adoptar para prevenir estas vulnerabilidades son:

  • Eliminar dependencias, funcionalidades, componentes, archivos y documentación innecesaria y no utilizada.
  • Utilizar una herramientas para inventariar las versiones de componentes. Como por ejemplo: Dependency Check y retire.js.
  • Monitorizar continuamente fuentes como CVE (Common Vulnerabilities and Exposures) y NVD (NIST - National Vulnerabilities Database) en búsqueda de vulnerabilidades en los componentes utilizados. Utilizar herramientas de análisis automatizados. Suscribirse a alertas de seguridad de los componentes utilizados.
  • Obtener componentes únicamente de orígenes oficiales utilizando canales seguros. Utilizar preferentemente paquetes firmados con el fin de reducir las probabilidades de uso de versiones manipuladas maliciosamente.
  • Supervisar bibliotecas y componentes que no poseen mantenimiento o no liberan parches de seguridad para sus versiones obsoletas o sin soporte.
