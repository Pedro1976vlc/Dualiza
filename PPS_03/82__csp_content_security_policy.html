8.2. - CSP (Content Security Policy)

La Política de Seguridad de Contenido (CSP) es un estándar de seguridad informática ideado para evitar o mitigar ataques que involucran inyección de contenido o código, como ataques de secuencias de comandos en sitios cruzados (XSS), ataques que requieren incrustar un recurso malicioso, ataques que involucran el uso malicioso de iframes, como ataques de clickjacking y otros. Este estándar permite a los desarrolladores controlar los lugares de origen desde donde  el navegador del cliente puede cargar recursos procedentes de otros sitios web.

Una política se describe utilizando una serie de directivas de políticas, cada una de las cuales describe la política para un determinado tipo de recurso o área de política.

Directivas y palabras clave CSP

CSP ofrece una gran cantidad de directivas configurables. A continuación se presentan algunas de las más importantes:

base-uri:

Define la URI (Uniform Resource Identifier, es una cadena que permite identificar la localización de un recurso) base para la URI relativa.

default-src:

Define la política de carga para todos los tipos de recursos en caso de que una directiva dedicada de tipo de recurso no esté definida.

script-src

Define qué scripts puede ejecutar el recurso protegido.

object-src:

Define desde dónde el recurso protegido puede cargar complementos.

style-src:

Define los orígenes desde donde se pueden cargar las hojas de estilos CSS. 

img-src:

Define desde dónde el recurso protegido puede cargar imágenes.

media-src:

Define desde dónde el recurso protegido puede cargar recursos multimedia: videos y audios.

child-src:

Define desde dónde el recurso protegido puede incrustar marcos.

Por ejemplo: child-src https://youtube.com habilitaría la incorporación de videos desde YouTube, pero no de otros orígenes. Emplee esta directiva en lugar de la obsoleta frame-src.

font-src:

Define desde dónde el recurso protegido puede cargar fuentes.

connect-src:

Define qué URI puede cargar el recurso protegido mediante interfaces de script.

manifest-src:

Define desde dónde el recurso protegido puede cargar manifiestos.

form-action:

Enumera los terminales para envíos a partir de etiquetas <form>.

form-ancestors:

Especifica los recursos que pueden incorporar a la página actual. Esta directiva se aplica a las etiquetas <frame>, <iframe>, <embed> y <applet>. Esta directiva no puede usarse en etiquetas <meta> y se aplica solo a recursos que no sean HTML.

upgrade-insecure-requests:

Indica a los usuario-agentes que modifiquen los esquemas de URL, cambiando HTTP por HTTPS. Esta directiva es para sitios web con grandes cantidades de URL antiguas que necesitan modificarse.

La lista de fuentes de contenido acepta cuatro palabras clave:

none

Como su nombre indica, que no coincida nada..

self

Coincide con la fuente actual, normalmente la propia web, pero no con sus subdominios.

unsafe-inline

Permite JavaScript y CSS incrustados procedentes de fuentes externas.

unsafe-eval

Permiten mecanismos texto-a-JavaScript como eval.

Estas directivas son solo algunos ejemplos. En cada nueva versión CSP va modificando e incorporando nuevas directivas. La lista completa y actualizada puede consultarse en la web específica del W3C sobre CSP.

Ejercicios Resueltos

A continuación, se presentan una serie de ejercicios prácticos resueltos mediante los cuales se ilustran la construcción de CSP.

Ejemplo 1:

Un sitio requiere que todo su contenido provenga de su propio dominio:

Retroalimentación

X-Content-Security-Policy: allow 'self'

Ejemplo 2:

Los administradores del servidor quieren rechazar todos los scripts de terceros para el sitio, y un grupo de proyecto determinado también quiere rechazar los medios de otros sitios (el encabezado proporcionado por los administradores de sistemas y el encabezado proporcionado por el grupo del proyecto están presentes):

Retroalimentación

X-Content-Security-Policy: allow *; script-src 'self' 
X-Content-Security-Policy: allow *; script-src 'self'; media-src 'self';

Ejemplo 3:

Un desarrollador web quiere protegerse su sitio contra los ataques de secuencias de comandos entre sitios. Pueden mitigar el riesgo de inyección de scripts asegurándose de que su CDN de confianza sea el único origen desde el que se puede cargar y ejecutar el script. Además, desean asegurarse de que ningún complemento pueda ejecutarse en los contextos de sus páginas. Especifique una política de seguridad que  tenga ese efecto:

Retroalimentación

Content-Security-Policy: script-src https://cdn.foo.com/scripts/; object-src 'none'
