5.2. - Estándares.

Muchas de las webs y aplicaciones que los usuarios utilizan requieren estar dados de alta en las mismas, siendo un problema recordar para todas ellas el usuario y la contraseña, además de poder tener varias cuentas en el mismo servicio y tener que salir de una para entrar en la otra. Estos problemas junto con la aparición de la necesidad de integración con aplicaciones de terceros, ya sean web o nativas han forzado la aparición de protocolos de autenticación y autorización que van más allá del uso de sesiones y “cookies” cómo el concepto “Single Sign-On” que persigue una única validación para un conjunto de aplicaciones web, todos ellos se basan en sistemas criptográficos más o menos complejos.

Descripción textual alternativa

Introducción a la criptografía de la UPV que explica los tipos de criptografía y la firma digital.

Los más destacados:

Certificados digitales.

Este es uno de los sistemas más seguros para autenticación. Normalmente es el servidor el que presenta su certificado al cliente, el cliente comprueba que dicho certificado está emitido por una entidad certificadora reconocida por el navegador (se puede comprobar en la configuración del navegador las entidades admitidas), pero en ocasiones se desea que el cliente se identifique con un certificado digital, pudiendo realizarse desde el propio servidor web  o desde los lenguajes. A pesar de ser uno de los sistemas más seguro es necesario disponer de un certificado emitido por una entidad certificadora de la que se fie el servidor o ola aplicación, además ha de estar instalado en el equipo desde el que se accede. Se recomienda para aplicaciones que necesiten una seguridad extrema.



En los vídeos anteriores de la Universidad de Murcia se explica los fundamentos, elementos y funcionamiento de los certificados digitales y como permite autenticar a personas y entidades.

Tokens

La autenticación y autorización con las sesiones y “cookies” obligan a disponer en el servidor de recursos en el que se almacenan las sesiones, estos recursos suelen ser memoria RAM. pudiendo ser problemático si existen muchos usuarios con sesiones abiertas como el número máximo de sesiones admitidas en el servidor, escalabilidad para encontrar el servidor en el que se encuentra la sesión del usuario o tener las sesiones replicadas en varios servidores.

 La validación con token se basa en que la información de la sesión pasa del servidor al cliente y en cada petición se envía el token. Por supuesto la información se encuentra encriptada y solo el servidor puede descodificar la información almacenada en dicho token.  

OAth.

Los token permiten interactuar entre cliente y servidor, y la información del usuario se encuentra en el servidor, por ejemplo en una interfaz web para correo: el usuario, la contraseña, la lista de contactos o los correos recibidos. Pero junto con el crecimiento web ha surgido la necesidad de comunicación e intercambio de información entre usuarios, aplicaciones en el que usuario posee cuenta y aplicaciones terceras en las que no tiene cuenta.

OpenId Connect.

Protocolo de autenticación con base OAuth 2.0 que añade una API Rest para obtener información del perfil del usuario y de cómo se autentico incluyendo quién, dónde, cuándo cómo, qué y  por qué.

Permite la autenticación federada de forma que es posible obtener información del usuario de un proveedor externo bien conocido  como Amazon, Facebook o Google.

Los mensajes se definen en JSON ofreciendo una API Rest sencilla que permite analizar dichos mensajes, además de poder integrarse en aplicaciones nativas de forma sencilla.
