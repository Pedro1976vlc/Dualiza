8. - Contramedidas. HSTS, CSP, CAPTCHAs, entre otros.

A lo largo de este tema se ha observado que muchos de los problemas de seguridad derivan del protocolo base utilizado: HTTP y de los diferentes métodos para conseguir transformar un protocolo sin estado a otro con estado en el que para conseguir la autenticación de uno de los extremos (el cliente, la autenticación del servidor se puede conseguir fácilmente con HTTPS) es necesario enviar las credenciales a través de la red, de la misma forma que para comprobar la autorización del cliente se ha de enviar información que permita identificarlo.

Si bien se han desarrollado protocolos que mejoran la seguridad del proceso como tokens o desinfección de datos enviados no resuelven todos los problemas de las aplicaciones web, algunos derivados de las tecnologías utilizadas en el cliente sobre HTTP. Las contramedidas son técnicas específicas contra amenazas, tratándose:

  • HSTS. (HTTP Strict Transport Security). HTTP es un protocolo inseguro que transmite la información en texto plano, si se consigue que la comunicación entre servidor y cliente pase de HTTPS a HTTP toda la información transmitida (usuarios, contraseñas, datos bancarios…) puede verse comprometida.
  • CSP (Content Security Policy). Evita principalmente los ataques XSS, el servidor indica al navegador las fuentes que son seguras, por ejemplo las URL desde los que se permite cargar ficheros Javascript o CSS, además puede forzar el protocolo para acceder esas URLs.
  • CAPTCHAs (Completely Automated Public Turing test to tell Computers and Humans Apart). Es relativamente sencillo crear programas o guiones que realicen peticiones masivas a servidores para romper claves por fuerza bruta o tramitar el alta masiva de cuentas. Los CAPTCHAs son retos con cierta complejidad que los programas no son capaces de solucionar o requieren de un alto coste computacional, sin embargo que para los humanos son sencillos.
  • CORS (Control de acceso HTTP). Las aplicaciones web han evolucionado y muchas de ellas utilizan servicios de otros sitios desde el cliente, al igual que un servidor puede tener que servir recursos a una web que no forma parte de su dominio  mediante por XMLRequest, por defecto no es posible realizar peticiones HTTP desde un cliente con una URL distinta a la del servidor que originó la web . Configurando en el servidor en caso de disponer de interfaces API es posible limitar el acceso a las URL seleccionadas. 
  • Subresource Integrity. Las aplicaciones web cargan recursos de otros sitios, aunque se ha de evitar en la medida de lo posible, utilizando recursos alojados en el propio servidor. Estos recursos pueden haberse visto comprometidos y modificados, cargando código malicioso. Para evitarlo se ha de añadir algún método que garantice que el recurso es válido y no se ha modificado. 
  • SameSite cookies. El almacén de “cookies” es común para todos los sitios web que se visitan, es decir, un sitio web puede solicitar cualquiera de las “cookies” existentes en el navegador, con SameSite se puede indicar que una cookie se restringirse a un contexto propio o del mismo sitio.
  • Otros: X-Content-Type-Options, X-Frame-Options, HPKP (similar a SSH pero se pasa la clave al cliente)
