5.2.1. - Tokens JWT

El estándar más extendido es JWT, que se basa en la aplicación de diferentes algoritmos como RS256 y ES51 siendo el más utilizado HMACSHA256. Este aplica al mensaje 2 dos veces la función hash SHA256 combinada con la clave secreta.

Descripción textual alternativa.

Base teórica de las funciones HASH o resumen, uno de los fundamentos de JWT, por la UPV.

Un token JWT independientemente del algoritmo utilizado se compone de 3 elementos expresados con JSON:

  • Cabecera: Indica el algoritmo de firma.
  • Cuerpo/payload/carga útil. Almacena la información a transmitir, dividiéndose a su vez en partes:
    • Registros recomendados pero no obligatoria del estándar como emisor (iss), tiempo de vencimiento(exp) entre otros.
    • Elementos públicos. Registros que se incluyen en función de la aplicación pero que han de ser los definidos en el estándar JWT de IANA, algunos ejemplos son profiel, website o phone_number.
    • Elementos privados: Se definen para cubrir necesidades no especificadas en los registros anteriores, por ejemplo el rol en una aplicación concreta.
  • Firma: Este elemento permite garantizar la identidad del emisor, ya que codifica los elementos anteriores con un algoritmo criptográfico (normalmente HMACSHA256, aunque existen muchos otros como RS256,ES512...). A este algoritmo se le facilita concatenados por . la cabecera codificada en base64Url, la carga útil codificada en base64url y el secreto. En cada petición HTTP, ya sea para código HTML, acceso a servidor REST o Web o vídeo el cliente envía el token al servidor en una cabecera del mensaje HTTP, se recomienda la cabecera Authorization que posee el siguiente formato:

 Authorization: <type> <credentials>

donde type puede tomar los valores: Basic,Bearer,Digest,HOBA ,AWS4, recomendado para JSW Bearer-
